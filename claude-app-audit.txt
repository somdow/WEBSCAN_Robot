                    HELLO WEB_SCANS — FULL APPLICATION AUDIT
                              February 18, 2026

OVERALL GRADE: 15-20% Bad (80-85% Solid)

The application is well-architected for its stage. The codebase follows Laravel
conventions, uses proper service-layer decomposition, has consistent naming,
and demonstrates thoughtful engineering decisions (gateway pattern for AI,
bot-protection fallback chains, signed URLs, etc.). The issues found are
typical of a fast-moving SaaS project that hasn't yet gone through a
hardening pass before public launch.

The "badness" is concentrated in a few specific areas (security edge cases,
missing rate limits, some oversized Blade files) rather than systemic
architectural rot. Nothing here is a "stop shipping" blocker, but the
Priority 1 items should be fixed before handling real user data.

================================================================================
                         PRIORITY 1 — SECURITY FIXES
================================================================================

1. MASS ASSIGNMENT ON USER MODEL
   File: app/Models/User.php
   Issue: `is_super_admin` and `deactivated_at` are in the $fillable array.
          If any controller passes unvalidated request data to User::create()
          or $user->update(), an attacker could escalate to super admin or
          reactivate a deactivated account.
   Fix:   Remove both from $fillable. Set them explicitly where needed:
          $user->is_super_admin = true; $user->save();

2. INVOICE DOWNLOAD IDOR
   File: app/Http/Controllers/BillingController.php (~line 228)
   Issue: downloadInvoice() takes an invoice ID but does not verify the
          invoice belongs to the authenticated user's organization. Any
          authenticated user could download another org's Stripe invoice
          by guessing/enumerating invoice IDs.
   Fix:   Validate that the invoice belongs to the current user's org
          before calling $organization->downloadInvoice($invoiceId).

3. ADMIN LOGIN VERIFICATION — MISSING RE-VALIDATION
   File: app/Http/Controllers/Auth/AdminLoginVerificationController.php
   Issue: After consuming the nonce, the controller calls Auth::loginUsingId()
          without checking if the user still exists, is still a super admin,
          and is still active (not deactivated). A user could be deactivated
          between email send and link click, and still gain access.
   Fix:   After retrieving the user_id from the nonce, load the user and
          verify: exists, isSuperAdmin(), deactivated_at is null.

4. FILAMENT ADMIN LOGIN — MISSING ACTIVE CHECK
   File: app/Filament/Pages/Auth/AdminLogin.php
   Issue: The authenticate() override checks isSuperAdmin() but does not
          check isActive() before sending the verification email. A
          deactivated admin could still trigger verification emails.
   Fix:   Add isActive() check before the verification flow.

================================================================================
                      PRIORITY 2 — RELIABILITY & ROBUSTNESS
================================================================================

5. QUEUE JOBS MISSING $timeout PROPERTY
   Files: app/Jobs/ProcessScanJob.php
          app/Jobs/ProcessPageAnalysisJob.php
          app/Jobs/DiscoverPagesJob.php
   Issue: All three jobs use retryUntil() (Carbon::now()->addMinutes(10))
          but none set a $timeout property. Without $timeout, a stuck job
          (e.g., waiting on a hung HTTP request) will never be killed by
          the queue worker — it just holds the worker slot until retryUntil
          expires, but retryUntil only prevents RE-queuing, it doesn't
          kill the running process.
   Fix:   Add `public $timeout = 540;` (9 minutes, under the 10-min retry
          window) to each job class.

6. MISSING RATE LIMITING ON RESOURCE-INTENSIVE ENDPOINTS
   Files: app/Http/Controllers/ScanController.php (store)
          app/Http/Controllers/AiController.php (optimize, reoptimize)
          app/Http/Controllers/PdfReportController.php (generate)
   Issue: Scan triggering, AI optimization, and PDF generation are expensive
          operations (API calls, queue jobs, memory-intensive rendering) with
          no rate limiting beyond plan-level credit checks. A malicious or
          buggy client could spam these endpoints.
   Fix:   Add throttle middleware to these routes. Suggested limits:
          - Scan creation: throttle:5,1 (5 per minute)
          - AI optimization: throttle:10,1
          - PDF generation: throttle:5,1

7. FOREIGN KEY CONSTRAINTS
   Files: Various migrations
   Issue: Some relationships rely on application-level enforcement only.
          If a project is deleted without cascading, orphaned scans,
          scan_pages, and module_results could remain. Most critical
          FKs are present, but worth auditing cascade behavior on:
          - scans.project_id -> projects.id
          - scan_pages.project_id -> projects.id
          - module_results.scan_id -> scans.id
   Fix:   Verify all FK constraints have appropriate onDelete behavior
          (cascade or set null as appropriate).

8. DATABASE INDEXES
   Files: Various migrations
   Issue: Columns frequently used in WHERE/ORDER BY clauses should be
          indexed. Candidates:
          - scans.status (filtered in dashboards and admin)
          - scans.project_id (already FK, likely indexed)
          - module_results.scan_id + module_results.module_key (compound)
          - scan_pages.project_id + scan_pages.analysis_status (compound)
   Fix:   Add indexes for high-frequency query patterns. Low priority
          while on SQLite, critical when migrating to MySQL at scale.

================================================================================
                    PRIORITY 3 — CODE QUALITY & MAINTAINABILITY
================================================================================

9. OVERSIZED BLADE COMPONENT: module-card.blade.php
   File: resources/views/components/scan/module-card.blade.php
   Issue: 479 lines in a single Blade component. Mixes AI panel logic,
          button states, score display, findings list, and recommendation
          rendering. Hard to maintain and test.
   Fix:   Decompose into sub-components:
          - <x-scan.module-card.ai-panel>
          - <x-scan.module-card.score-ring>
          - <x-scan.module-card.findings-list>
          - <x-scan.module-card.action-buttons>
          Target: each sub-component under 120 lines.

10. OVERSIZED BLADE VIEW: projects/show.blade.php
    File: resources/views/projects/show.blade.php
    Issue: The combined project+scan view is feature-rich but large.
           Multiple tab panels, score sections, accordion groups, page
           management, and scan history all in one template.
    Fix:   Already partially addressed with 8 partials in scans/partials/.
           Consider extracting the tab navigation and page management
           sections into their own partials.

11. SERVICE CLASS SIZE
    File: app/Services/ScanOrchestrator.php
    Issue: The orchestrator handles both single-page and crawl scan flows,
           bot detection, Zyte fallback, screenshot capture, and progress
           tracking. While well-organized with clear method boundaries,
           it's approaching the complexity threshold.
    Fix:   Not urgent — the branching (executeSinglePageScan vs
           executeCrawlScan) is clean. Monitor as features are added.
           If it grows past ~400 lines, consider extracting a
           BotDetectionService or ScreenshotCaptureService.

12. INCONSISTENT ARRAY SYNTAX IN VENDOR-TOUCHED FILES
    Issue: Project standard is array() syntax, but some files generated
           by Breeze/Filament use [] syntax. Not a bug, but inconsistent.
    Fix:   Low priority. Normalize during routine maintenance if desired.
           Focus on project-authored files only.

13. MODEL CASTS
    File: app/Models/User.php, app/Models/Scan.php
    Issue: Some boolean/datetime columns could benefit from explicit casts
           to avoid type juggling bugs:
           - User: is_super_admin -> boolean cast
           - User: deactivated_at -> datetime cast
           - Scan: progress_percent -> integer cast
    Fix:   Add to $casts array on respective models.

================================================================================
                      PRIORITY 4 — MINOR / NICE-TO-HAVE
================================================================================

14. SETTINGS TABLE AS CONFIG STORE
    Issue: Zyte API key and enabled flag are stored in the settings table
           rather than .env/config. This works and enables admin UI
           management, but means these values aren't available during
           config caching (php artisan config:cache).
    Fix:   Not necessarily a problem — the Setting::getValue() pattern
           queries at runtime. Just be aware that config:cache won't
           include these values. Document this behavior.

15. ERROR PAGE CUSTOMIZATION
    Issue: Default Laravel error pages (404, 500, 403, 419) are still
           the framework defaults. For a SaaS product, branded error
           pages improve user experience.
    Fix:   Create custom error views in resources/views/errors/ when
           preparing for public launch.

16. CSRF TOKEN EXPIRATION UX
    Issue: Long-idle sessions will get 419 errors on form submission.
           No client-side handling to detect expired sessions and
           prompt re-login.
    Fix:   Add a meta tag with CSRF token + periodic refresh via
           Alpine.js, or catch 419 responses in fetch handlers and
           show a "session expired" modal.

17. QUEUE FAILURE NOTIFICATIONS
    Issue: No notification system for failed queue jobs. If a scan job
           fails silently, the user sees a stuck progress bar but admins
           aren't alerted.
    Fix:   Implement a failed job handler that logs to a monitoring
           channel (email, Slack, or admin dashboard widget).

================================================================================
                          WHAT'S DONE WELL
================================================================================

- Clean service-layer architecture: Controllers are thin, business logic
  lives in dedicated services (ScanOrchestrator, PageAdditionService,
  BillingService, AdminLoginVerificationService, etc.)

- Gateway pattern for AI providers: 3 providers (OpenAI, Anthropic, Google)
  behind a unified interface. Easy to add/swap providers.

- Bot protection chain: Guzzle -> validate -> Zyte fallback -> validate ->
  mark blocked. Graceful degradation with clear status tracking.

- Centralized UI config: scan-ui.php config file for sidebar groups,
  category icons, score ring parameters. DRY and maintainable.

- Proper use of Laravel features: signed URLs, queued notifications,
  cache nonces, middleware groups, form requests, policies.

- Security consciousness: CSRF on all forms, prepared statements via
  Eloquent, output escaping in Blade, signed routes for sensitive actions,
  throttle middleware on auth routes.

- Modular analyzer architecture: 48 modules across 10 categories with
  clear scope classification (site-wide vs per-page). Easy to add new
  analyzers without touching orchestration code.

- Consistent naming and code style across project-authored files.

- Good separation of concerns: Models, Services, Controllers, Jobs,
  Notifications each in their proper namespace.

- Reusable Blade components: accordion-section, progress-bar, module-card,
  AI overlay — all extracted and parameterized.
